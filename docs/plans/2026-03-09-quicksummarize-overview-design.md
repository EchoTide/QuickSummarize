# QuickSummarize 概要设计说明书

## 1 引言

### 1.1 编写目的
本文档依据 GB/T 8567 软件文档编制的通用体例，结合 QuickSummarize 项目当前实现情况，给出系统级概要设计方案，为后续详细设计、编码实现、测试验证和部署运维提供统一依据。

### 1.2 项目背景
QuickSummarize 是一个面向 YouTube 视频场景的智能摘要系统，当前以 Chrome 扩展为主要运行载体，提供视频信息识别、字幕抓取、AI 实时总结、分段时间线总结和中英文界面切换能力；同时配套一个基于 Next.js 的产品官网用于功能展示与下载引导。

### 1.3 术语和缩略语
- AI：人工智能。
- LLM：大语言模型。
- SSE：Server-Sent Events，服务端事件流。
- MV3：Chrome Extension Manifest V3。
- Side Panel：Chrome 浏览器侧边栏扩展界面。
- timedtext：YouTube 字幕接口返回数据。

## 2 系统概述

### 2.1 建设目标
系统目标是在不依赖专有后端摘要服务的前提下，为用户提供面向 YouTube 视频的轻量化智能摘要能力。系统通过浏览器扩展直接感知视频页面、提取字幕内容、调用兼容 OpenAI 协议的模型接口生成结构化摘要，并以侧边栏和页面嵌入面板两种方式输出结果。

### 2.2 设计范围
本概要设计覆盖以下内容：
- Chrome 扩展端总体结构与模块划分。
- 字幕获取、摘要生成、时间线总结的主要处理流程。
- 本地配置存储、扩展消息通信和页面桥接机制。
- Web 官网子系统的功能定位与部署方式。
- 外部接口、运行环境、安全约束及后续扩展点。

以下内容不在当前设计范围内：
- 自建服务端摘要引擎实现。
- 用户账号体系、计费体系与云端配置同步。
- 多浏览器兼容性适配方案。

### 2.3 设计原则
- 以扩展本地执行为主，降低服务端依赖。
- 以模块解耦为原则，按字幕、模型、存储、页面识别等能力拆分。
- 以可验证为原则，核心处理逻辑具备测试覆盖。
- 以渐进增强为原则，Web API 作为预留能力，不影响主链路可用性。

## 3 总体设计

### 3.1 总体架构
系统采用“浏览器扩展主导、官网展示辅助、第三方模型服务外部提供”的总体架构，逻辑上分为四层：

1. 表现层：由 Side Panel、页面内嵌 iframe 面板和官网页面组成，负责与用户交互。
2. 业务控制层：由内容脚本、后台脚本和侧边栏控制逻辑组成，负责视频识别、状态流转、消息调度和异常处理。
3. 领域能力层：由字幕解析、字幕合并、时间线总结、语言切换、存储和模型调用等库组成。
4. 外部依赖层：包括 YouTube 页面及其字幕数据、兼容 OpenAI 协议的模型服务、Chrome 扩展运行环境。

### 3.2 子系统划分

#### 3.2.1 Chrome 扩展子系统
Chrome 扩展子系统是系统核心，包含以下模块：
- 后台模块：负责打开 Side Panel、执行跨域代理请求、转发模型流式响应。
- 内容脚本模块：负责感知 YouTube 页面、注入页面钩子、收集视频信息、承接字幕请求与页面控制。
- 页面钩子模块：运行于页面上下文，拦截 `fetch/XMLHttpRequest`，捕获字幕数据，管理字幕轨道切换与预热。
- Side Panel 模块：负责摘要生成、分段总结展示、复制、重试、取消、跳转视频时间点等交互。
- 配置与工具库模块：负责存储、国际化、时间线处理、字幕合并、视频信息标准化等通用能力。

#### 3.2.2 Web 官网子系统
Web 官网子系统承担产品介绍、下载引导和中英文展示职责，当前不参与实际摘要链路。`/api/summarize` 接口处于占位状态，返回 `501 Not implemented yet`，因此不作为正式业务接口使用。

### 3.3 部署形态
- 扩展端通过根目录 `npm run build` 构建，将脚本产物输出到 `extension/dist`，以开发者模式加载到 Chrome。
- 官网端在 `web` 目录下执行 `npm run build` 与 `npm run start` 进行标准 Next.js 部署。
- 模型服务由用户自行配置第三方兼容 OpenAI 协议的接口地址、模型名称和 API Key。

## 4 模块设计

### 4.1 后台模块设计
后台模块由 `extension/background.js` 实现，主要职责如下：
- 响应浏览器扩展图标点击行为，自动打开侧边栏界面。
- 接收前端模块的 `PROXY_FETCH` 请求并发起网络访问。
- 基于 `chrome.runtime.connect` 建立 `QS_SSE_PROXY` 通道，承接模型流式响应并将 SSE 文本分片返回给 Side Panel。
- 对 YouTube `timedtext` 请求做代理限制，避免字幕获取链路偏离页面上下文拦截机制。

后台模块不承担业务编排，只提供扩展权限环境下的公共基础能力。

### 4.2 内容脚本模块设计
内容脚本由 `extension/content.js` 实现，主要职责如下：
- 识别当前标签页是否为 YouTube 视频页，并提取视频编号与标题。
- 动态注入 `page-hook.js` 进入页面上下文，以便获取字幕轨道和 timedtext 响应。
- 监听单页应用路由变化、历史记录变化、页面事件与定时轮询，保证视频切换后信息同步。
- 对 Side Panel 提供 `REQUEST_TRANSCRIPT_V2`、`REQUEST_VIDEO_INFO`、`SEEK_TO` 等消息服务。
- 创建页面内嵌摘要面板，并通过 `postMessage` 与 iframe 保持高度联动。

### 4.3 页面钩子模块设计
页面钩子由 `extension/page-hook.js` 实现，运行于 YouTube 页面上下文，主要职责如下：
- Hook 页面 `fetch` 和 `XMLHttpRequest`，捕获 `/api/timedtext` 响应。
- 从播放器对象中提取字幕轨道 URL，支持获取不同语言字幕轨道。
- 支持字幕语言切换、字幕预热和字幕数据回传。
- 通过 `window.postMessage` 向内容脚本发送 `TIMEDTEXT_RESPONSE`、`CAPTION_TRACK_URLS` 等桥接消息。

该模块是系统字幕获取能力的关键模块，决定了系统对 YouTube 页面变化的适配能力。

### 4.4 Side Panel 模块设计
Side Panel 由 `extension/sidepanel.js` 实现，是系统主要交互控制器，采用状态驱动方式组织界面，主要状态包括：未配置、无视频、就绪、加载中、已完成、分段总结、错误。

该模块主要职责如下：
- 加载和保存用户配置。
- 请求当前视频信息并维护视频上下文。
- 发起字幕请求并维护字幕缓存。
- 调用模型服务进行实时摘要生成。
- 调用时间线总结能力生成分段摘要。
- 将 Markdown 格式内容渲染到界面中。
- 支持取消、重试、复制、跳转到视频时间点等交互。

### 4.5 存储模块设计
存储模块由 `extension/lib/storage.js` 实现，基于 `chrome.storage.local` 保存本地配置，当前数据项包括：
- `baseUrl`：模型服务基础地址。
- `model`：模型标识。
- `apiKey`：调用凭据。
- `language`：界面和摘要语言，默认值为 `en`。

当前设计不提供云端同步与加密存储能力。

### 4.6 模型调用模块设计
模型调用模块由 `extension/lib/llm.js` 实现，负责构造兼容 OpenAI 协议的请求报文、解析 SSE 流式返回、清理推理痕迹文本，并根据语言选取不同系统提示词。

该模块的主要设计点如下：
- 统一构造 `/chat/completions` 请求体。
- 默认启用 `stream: true` 实现实时输出。
- 优先使用运行时端口代理流式请求，失败时降级为消息代理或直接 `fetch`。
- 解析 SSE 行格式，抽取 `delta.content` 作为可展示文本。
- 对中文输出场景执行额外清理，尽量屏蔽英文思维链与无关前缀。

### 4.7 字幕与时间线处理模块设计
字幕与时间线处理能力位于 `extension/lib/transcript.js`、`extension/lib/subtitles.js`、`extension/lib/timeline-summary.js`，职责如下：
- 解析 YouTube 字幕原始响应为带时间戳的字幕片段集合。
- 将细粒度字幕片段合并为更适合阅读和总结的段落。
- 对长字幕按块分批生成时间线摘要。
- 约束模型以 JSON 形式返回分段结果，并在失败时执行修正或降级处理。

### 4.8 国际化与辅助模块设计
国际化模块由 `extension/lib/i18n.js` 和官网端 `web/src/context/LangContext.tsx`、`web/src/i18n/*` 构成。扩展端和官网端均支持中英文切换，但状态持久化方式不同：扩展端写入本地配置，官网端当前主要在前端上下文中维护。

辅助模块还包括：
- `video-page.js`：负责视频 URL 识别。
- `video-sync.js`：负责视频信息标准化及变化检测。
- `embed-resize.js`：负责嵌入面板高度测量。

## 5 接口设计

### 5.1 内部消息接口
系统内部主要采用 Chrome 扩展消息机制和页面桥接机制。

#### 5.1.1 扩展内部消息
典型消息如下：
- `VIDEO_DETECTED`：视频识别事件。
- `REQUEST_TRANSCRIPT_V2`：请求字幕内容及结构化片段。
- `REQUEST_VIDEO_INFO`：请求当前视频信息。
- `SEEK_TO`：控制视频跳转到指定时间点。
- `PROXY_FETCH`：请求后台代为访问外部接口。

#### 5.1.2 页面桥接消息
内容脚本与页面钩子之间通过 `window.postMessage` 传递消息，典型消息如下：
- `REQUEST_CAPTION_TRACKS`
- `CAPTION_TRACK_URLS`
- `PREFETCH_TIMEDTEXT`
- `TIMEDTEXT_RESPONSE`
- `SWITCH_CAPTION_LANGUAGE`

### 5.2 外部接口设计

#### 5.2.1 模型服务接口
- 调用方式：HTTP POST。
- 目标路径：`{baseUrl}/chat/completions`。
- 协议特点：兼容 OpenAI 风格请求；支持 SSE 流式响应。
- 核心输入：模型名称、系统提示词、字幕文本。
- 核心输出：结构化摘要文本或分段总结 JSON。

#### 5.2.2 Web API 接口
`web/src/app/api/summarize/route.ts` 当前仅返回未实现提示，设计上可作为未来服务端汇聚接口预留，但在当前版本中不纳入正式依赖。

## 6 数据设计

### 6.1 主要数据对象

#### 6.1.1 系统配置对象
配置对象结构如下：

```json
{
  "baseUrl": "string",
  "model": "string",
  "apiKey": "string",
  "language": "en|zh"
}
```

#### 6.1.2 视频信息对象

```json
{
  "videoId": "string",
  "title": "string"
}
```

#### 6.1.3 字幕片段对象

```json
{
  "startSec": 0,
  "text": "string"
}
```

#### 6.1.4 字幕缓存对象
Side Panel 内维护字幕缓存，包含视频标识、语言、完整字幕文本、原始片段、合并片段以及按语言区分的分段总结结果集合。

### 6.2 数据存储策略
- 本地持久数据存储在 `chrome.storage.local`。
- 运行期缓存数据主要保存在内容脚本和 Side Panel 内存中。
- 官网当前无独立业务数据库。

## 7 运行设计

### 7.1 主业务流程
系统主业务流程如下：

1. 用户打开 YouTube 视频页。
2. 内容脚本识别视频并注入页面钩子。
3. 页面钩子获取字幕轨道或拦截字幕响应。
4. Side Panel 请求视频信息与字幕内容。
5. 字幕解析模块生成文本和片段集合。
6. 模型调用模块将字幕发送到外部模型服务。
7. 后台模块代理流式返回，Side Panel 实时展示摘要结果。
8. 若用户查看分段总结，则时间线模块按片段批量生成摘要并支持点击跳转。

### 7.2 异常处理设计
系统对主要异常场景做统一处理：
- 未配置模型参数时，进入未配置状态并引导用户打开设置页。
- 未打开 YouTube 视频页时，进入无视频状态。
- 无字幕、字幕为空或字幕抓取失败时，显示错误信息并允许重试。
- 模型调用失败或用户主动取消时，转入错误或取消状态。
- 流式输出异常时，允许重新发起请求。

### 7.3 性能设计考虑
- 采用流式输出减少长文本生成等待感。
- 通过字幕缓存、轨道 URL 缓存和预热机制减少重复抓取。
- 通过分段总结方式控制超长字幕的模型输入规模。
- 通过页面内嵌面板高度自适应降低界面抖动。

## 8 安全设计

### 8.1 现状说明
当前系统安全设计以本地扩展运行安全为主，尚未形成完整的服务端安全体系。主要特点如下：
- API Key 保存在本地浏览器存储中，未加密。
- 扩展申请了较宽的主机访问权限，用于兼容不同模型服务地址。
- 页面钩子会修改页面上下文中的网络请求行为，但作用域限定在 YouTube 页面。
- Side Panel 使用 Markdown 渲染模型返回内容，当前未见独立 HTML 清洗链路。

### 8.2 风险与约束
- 本地敏感凭据泄露风险需要通过浏览器本地环境安全控制。
- 广泛主机权限需要在发布与审计阶段进行严格评估。
- 模型输出内容可能带来渲染型安全风险，建议后续增加内容净化措施。
- 页面结构或字幕接口策略变化可能导致字幕抓取失效。

## 9 Web 官网设计说明

### 9.1 功能定位
官网用于产品展示、功能介绍、使用说明和下载引导，不承担扩展主链路中的字幕处理和摘要生成职责。

### 9.2 页面组成
官网首页由 `Header`、`Hero`、`Features`、`HowItWorks`、`Download`、`Footer` 等组件构成，并通过语言上下文切换中英文展示。

### 9.3 后续扩展方向
后续可将官网扩展为统一配置入口、服务端摘要中转入口、版本发布页或用户帮助中心，但应在详细设计阶段补充账号、鉴权、限流、日志和运维方案。

## 10 约束与待完善项

### 10.1 技术约束
- 当前系统依赖 Chrome 扩展 MV3 运行环境。
- 字幕获取能力依赖 YouTube 页面结构和字幕机制。
- 模型服务能力依赖用户自行配置的第三方接口质量。

### 10.2 待完善项
- 补充服务端摘要接口正式实现与鉴权机制。
- 增加 API Key 加密或更安全的托管方案。
- 完善模型输出内容净化与安全防护。
- 增加更细粒度的日志、监控和错误诊断能力。
- 补充官网与扩展之间的统一版本和配置协同机制。

## 11 结论
QuickSummarize 当前版本已形成较完整的浏览器扩展式智能摘要架构，其核心特征是：以 Chrome 扩展为执行主体、以 YouTube 页面字幕捕获为数据来源、以第三方兼容 OpenAI 协议的模型服务为智能能力来源、以官网为展示和下载辅助入口。该架构实现简单、部署轻量、依赖清晰，适合作为后续详细设计和产品化增强的基础版本。
